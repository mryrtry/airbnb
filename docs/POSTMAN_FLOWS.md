# Флоу и приколы API — как повторять сценарии по коллекции

Коллекция: **`docs/airbnb-flows.postman_collection.json`**. Импорт: Postman → File → Import → выберите файл.

Переменные коллекции заполняются автоматически после Login/Register и после Create listing / Create booking / Open resolution. Для прода подставьте `baseUrl` (например `http://se.ifmo.ru:8126` или через туннель).

---

## Переменные

| Переменная     | Откуда берётся |
|----------------|----------------|
| `baseUrl`      | По умолчанию `http://localhost:9000` |
| `accessToken`  | После **Login** или **Register** (скрипт в Tests) |
| `refreshToken` | То же |
| `listingId`    | После **Create listing** |
| `bookingId`    | После **Create booking** |
| `resolutionId` | После **Open resolution for booking** или **Get resolution by booking id** (подставить вручную) |
| `notificationId` | Вручную из ответа **Get all notifications** |
| `userId`       | Вручную из ответа **Get all users** |

Важно: при смене роли всегда делайте нужный **Login** — следующий запрос пойдёт с новым токеном.

---

## 1. Флоу: от регистрации до выезда (полный цикл бронирования)

| # | Действие | Запрос в коллекции | Кто может | Кто не может | Что ожидать |
|---|----------|---------------------|-----------|--------------|-------------|
| 1 | Проверка сервиса | Health → Health check | Все | — | 200, тело с `status` |
| 2 | Регистрация владельца | Auth → Register (owner) | Любой (публично) | — | 201, в переменные пишутся токены |
| 3 | Создать объявление | Listings → Create listing | Владелец (LISTING_CREATE) | Гость без роли владельца | 201, `listingId` в переменные; статус DRAFT |
| 4 | Опубликовать объявление | Listings → Update listing (publish) | Владелец **этого** листинга или админ | Другой пользователь | 200, статус PUBLISHED; владельцу уходит LISTING_PUBLISHED |
| 5 | Регистрация гостя | Auth → Register (guest) | Любой | — | 201, токены в переменные |
| 6 | Заявка на бронирование | Bookings → Create booking | Гость (BOOKING_CREATE) | Владелец тем же запросом не создаёт «от своего имени» | 201, `bookingId` в переменные. Тело: `listingId`, `checkInDate`, `checkOutDate` (ISO). Листинг PUBLISHED. Дата выезда > даты въезда. Один гость не может иметь две заявки с пересекающимися датами по одному объявлению (400). |
| 7 | Войти владельцем | Auth → Login (owner) | — | — | 200, токен владельца в переменные |
| 8 | Одобрить заявку | Bookings → Decide booking (approve) | Владелец **этого** листинга или админ | Гость, другой владелец | 200, статус APPROVED. Все остальные заявки (APPLIED) по этому объявлению с пересекающимися датами автоматически отклоняются; подавшим уходит BOOKING_REJECTED_ANOTHER_APPROVED. Если на эти даты уже одобрена/занята другая бронь — 400. |
| 9 | Войти гостем | Auth → Login (guest) | — | — | 200, токен гостя |
| 10 | Въезд (check-in) | Bookings → Check-in | **Только гость этого бронирования или админ** | Владелец не может | 200, статус CHECKED_IN |
| 11 | Выезд (check-out) | Bookings → Check-out | **Только гость этого бронирования или админ** | Владелец не может | 200, статус CHECKED_OUT; может автоматически открыться окно разрешения споров |

**Прикол:** Check-in и Check-out дергает только гость бронирования или админ. Владелец получит 403 (CANNOT_ACCESS_SOURCE), даже если у него есть право BOOKING_READ.

**Даты бронирования:**
- У каждой заявки есть **дата въезда** (`checkInDate`) и **дата выезда** (`checkOutDate`). В дату выезда первого может заехать второй (два человека не живут в одну дату в одном объявлении).
- **Один гость** не может создать больше одной заявки на одно объявление с пересекающимися датами (400: BOOKING_GUEST_OVERLAPPING_DATES).
- **Много заявок** на одни и те же даты по одному объявлению разрешены. Когда владелец **одобряет одну**, все остальные заявки (APPLIED) с пересекающимися датами **автоматически отклоняются**, подавшим приходит уведомление BOOKING_REJECTED_ANOTHER_APPROVED.
- Одобрить вторую заявку на уже занятые даты нельзя (400: BOOKING_LISTING_OCCUPIED_DATES).

## 2. Флоу: окно разрешения споров — гость платит

Окно уже открыто (после check-out) или открывается вручную. Владелец запрашивает деньги, гость соглашается.

| # | Действие | Запрос | Кто может | Кто не может | Что ожидать |
|---|----------|--------|-----------|--------------|-------------|
| 1 | Войти владельцем | Auth → Login (owner) | — | — | Токен владельца |
| 2 | Открыть окно (если ещё нет) | Resolutions → Open resolution for booking | Владелец/гость этого бронирования или админ | Чужой пользователь | 201, `resolutionId` в переменные; статус OPEN |
| 3 | Запросить компенсацию | Resolutions → Request money | **Владелец** этого окна (или админ) | Гость | 200, статус MONEY_REQUESTED; гостю уходит RESOLUTION_MONEY_REQUESTED |
| 4 | Войти гостем | Auth → Login (guest) | — | — | Токен гостя |
| 5 | Согласие оплатить | Resolutions → Respond pay | **Гость** этого окна (или админ) | Владелец | 200, окно закрывается; владельцу уходит уведомление об оплате |

**Прикол:** Request money только из статуса OPEN. Respond pay допустим из MONEY_REQUESTED и из MANDATORY_PAYMENT (после эскалации и решения «обязательная выплата»).

---

## 3. Флоу: гость отказал → владелец эскалирует → вмешательство Airbnb

Гость отвечает «отказываюсь», владелец просит вмешательства, админ решает: обязательная выплата или урегулирование без взыскания.

| # | Действие | Запрос | Кто может | Кто не может | Что ожидать |
|---|----------|--------|-----------|--------------|-------------|
| 1–3 | Как в флоу 2 (владелец открыл окно, запросил деньги) | Login (owner), Open resolution, Request money | — | — | Окно в MONEY_REQUESTED |
| 4 | Войти гостем | Auth → Login (guest) | — | — | — |
| 5 | Отказ платить | Resolutions → Respond refuse | **Гость** этого окна (или админ) | Владелец | 200, статус REFUSED |
| 6 | Войти владельцем | Auth → Login (owner) | — | — | — |
| 7 | Эскалация | Resolutions → Escalate | **Владелец** этого окна (или админ) | Гость | 200, статус ESCALATED; гостю уходит RESOLUTION_ESCALATED. Эскалация только из REFUSED |
| 8 | Войти админом | Auth → Login (admin) | — | — | admin / password |
| 9a | Вмешательство: обязательная выплата | Resolutions → Вмешательство Airbnb → **Обязательная выплата гостю (admin)** | **Только админ** (ALL_RESOLUTION_UPDATE) | Владелец, гость | 200, статус MANDATORY_PAYMENT; гостю RESOLUTION_MANDATORY_PAYMENT_ASSIGNED. Дальше гость может **Respond pay** |
| 9b | Вмешательство: урегулировать без взыскания | Resolutions → Вмешательство Airbnb → **Урегулировать конфликт без взыскания (admin)** | **Только админ** | Владелец, гость | 200, статус CLOSED; владельцу RESOLUTION_RESOLVED_WITHOUT_PAYMENT |

**Прикол:** Resolve escalation дергает **только админ**. Владелец и гость получат 403.

**Прикол:** Если после 9a гость всё-таки платит: снова **Login (guest)** → **Respond pay** — тогда окно закрывается (оплата по обязательной выплате).

---

## 4. Флоу: владелец закрывает окно без эскалации (после отказа гостя)

Гость отказал (REFUSED), владелец решает не эскалировать, а просто закрыть окно.

| # | Действие | Запрос | Кто может | Кто не может | Что ожидать |
|---|----------|--------|-----------|--------------|-------------|
| 1–5 | Как в флоу 3 до Refuse | … → Respond refuse | — | — | Статус REFUSED |
| 6 | Закрыть без эскалации | Resolutions → Close resolution (admin or owner from REFUSED) | **Владелец этого окна** (только из REFUSED) или **админ** (из любого статуса) | Гость | 200, статус CLOSED; владельцу уходит RESOLUTION_WINDOW_CLOSED_BY_OWNER |

**Прикол:** Владелец может закрыть окно только если статус REFUSED. Из OPEN/MONEY_REQUESTED владелец закрыть не может — только админ.

---

## 5. Флоу: жалоба без запроса денег

Владелец из открытого окна подаёт жалобу (без суммы). Окно закрывается.

| # | Действие | Запрос | Кто может | Кто не может | Что ожидать |
|---|----------|--------|-----------|--------------|-------------|
| 1–2 | Владелец, открытое окно | Login (owner), Open resolution | — | — | Статус OPEN |
| 3 | Подать жалобу | Resolutions → Complaint | **Владелец** (или админ) | Гость | 200, статус COMPLAINT_RECORDED, окно закрыто; владельцу уходит RESOLUTION_COMPLAINT_RECORDED |

**Прикол:** Complaint только из статуса OPEN. Если уже запросили деньги (MONEY_REQUESTED), жалобу этим эндпоинтом не подать.

---

## 6. Флоу: админ закрывает окно

Админ может закрыть любое окно в любой момент (кроме уже закрытых).

| # | Действие | Запрос | Кто может | Кто не может | Что ожидать |
|---|----------|--------|-----------|--------------|-------------|
| 1 | Войти админом | Auth → Login (admin) | — | — | — |
| 2 | Закрыть окно | Resolutions → Close resolution (admin or owner from REFUSED) | **Только админ** (из любого статуса) или владелец из REFUSED | Гость, владелец из OPEN | 200, CLOSED; владельцу RESOLUTION_WINDOW_CLOSED |

---

## 7. Уведомления и пользователи

- **Get all notifications / Get unread count / Mark read / Mark all read** — любой авторизованный пользователь видит **только свои** уведомления (NOTIFICATION_READ или ALL_NOTIFICATION_READ).
- **Get me / Get permissions** — любой авторизованный.
- **Get all users / Get user by id / Update user / Update roles / Delete user** — только админ (ALL_USER_READ / ALL_USER_UPDATE / ALL_USER_DELETE).

---

## Сводка: кто какую ручку дергает

| Ручка | Кто может | Кто не может |
|-------|-----------|--------------|
| Check-in / Check-out | Гость **этого** бронирования или админ | Владелец |
| Request money / Escalate / Complaint | Владелец окна (листинга бронирования) или админ | Гость |
| Respond pay / Respond refuse | Гость окна или админ | Владелец |
| Close resolution | Админ всегда; владелец — только из REFUSED | Гость; владелец из OPEN |
| Resolve escalation | Только админ | Владелец, гость |
| Create listing / Update / Delete listing | Владелец **этого** листинга или админ | Другой пользователь |
| Create booking | Любой с BOOKING_CREATE (гость). Тело: listingId, checkInDate, checkOutDate | Пересекающиеся даты у того же гостя по тому же объявлению → 400 |
| Decide (approve) | Владелец листинга или админ | Если на эти даты уже APPROVED/CHECKED_IN → 400; при успехе остальные APPLIED с теми же датами отклоняются и получают уведомление |

---

## Ожидаемые коды ответа

- **200/201** — успех (тело по контракту).
- **400** — невалидное тело (валидация); даты бронирования (выезд ≤ въезд, пересечение у одного гостя, занятые даты при одобрении).
- **403** — нет права или доступ к ресурсу запрещён (CANNOT_ACCESS_SOURCE).
- **404** — ресурс не найден или неверный статус для действия (SOURCE_WITH_ID_NOT_FOUND и т.п.).

При 403/404 в теле обычно приходит сообщение об ошибке (например, «Cannot access source: Booking» для check-in владельцем).
